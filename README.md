## provider.tf 생성
-- provider == 제공자
AWS 같은 클라우드는 전 세계에 데이터 센터가 퍼져 있습니다(서울 리전, 도쿄 리전, 버지니아 리전 등). 
테라폼에게 단순히 "ALB 하나 만들어줘"라고만 하면, 테라폼은 "그래서 지구상 어디에 만들라는 거야?"라고 되묻게 된다.
이때 provider.tf가 그 답을 주는 곳입니다.

<provider 예시>
# 1. 어떤 프로바이더(제공자)를 쓸 것인가?
provider "aws" {
  # 2. 어느 리전(지역)에 인프라를 구축할 것인가?
  region = "ap-northeast-2" # 여기가 바로 '서울 리전'을 쓰겠다고 지정하는 부분!
}


## main.tf 생성
<VPC 공부>
쉽게 말하면, 거대한 AWS라는 공용 클라우드 공간 안에 **"나만 사용할 수 있는 가상의 개인 네트워크 망"**을 따로 떼어놓은 것이라고 생각하면 됩니다.

1. VPC를 왜 쓰나요? (비유로 이해하기)
AWS 전체를 **'커다란 공용 운동장'**이라고 한다면, VPC는 그 안에 **'나만의 울타리를 치고 만든 개인 구역'**입니다.

VPC가 없다면: 누구나 내 서버에 쉽게 접근할 수 있고, 보안 설정이 매우 복잡해집니다.

VPC가 있다면: 울타리 안에 누가 들어올지(인터넷 연결 여부), 어떤 서버끼리 대화할지를 내가 완전히 통제할 수 있습니다.

2. VPC의 핵심 구성 요소
VPC를 제대로 이해하려면 함께 나오는 단어 몇 가지만 알면 됩니다.

서브넷(Subnet): VPC라는 커다란 땅을 다시 작은 구역으로 나눈 것입니다.

Public Subnet: 인터넷과 연결되어 외부에서 접속 가능한 구역 (예: 웹 서버, ALB)

Private Subnet: 인터넷과 직접 연결되지 않은 안전한 구역 (예: DB 서버)

IP 대역 (CIDR): 이 네트워크 안에서 사용할 주소 범위를 정합니다. (예: 10.0.0.0/16)

라우팅 테이블(Route Table): 트래픽이 어디로 가야 할지 알려주는 '이정표'입니다.

인터넷 게이트웨이(IGW): VPC를 외부 인터넷과 연결해주는 '대문' 역할을 합니다.
